# 多模态大语言模型中的Token剪枝：我们在解决正确的问题吗

## Token Pruning in Multimodal LLMs: Are We Solving the Right Problem?

**作者**: Yike Guo, Xin Wang, Wenwu Wang, Yu Li, Jin Liu, Hongxia Li
**机构**: 多所知名大学联合研究
**发表**: EMNLP 2024
**arXiv**: 2409.04157

---

## 摘要

多模态大语言模型（MLLMs）的Token剪枝技术已成为提升推理效率的关键研究方向。然而，现有研究在评估方法和实验设计上存在诸多不一致性，导致难以准确评估各方法的有效性。本文对当前Token剪枝方法进行了系统性批判分析，揭示了三个核心问题：**评估指标的不一致性**——延迟、FLOPs和吞吐量等指标往往给出相互矛盾的结论；**剪枝决策的任务相关性**——在某一任务上表现良好的方法可能在其他任务上效果欠佳；**比较基准的不统一性**——缺乏标准化的评估流程使得不同方法难以公平比较。更重要的是，我们发现许多Token剪枝方法实际上并未显著减少实际推理延迟，部分方法的加速效果甚至不如简单地使用更小的批处理大小。通过这些发现，我们提出了一套更全面的评估框架，包括多维度指标评估、任务泛化性测试和标准化比较流程。最后，我们指出了该领域未来研究的正确方向：任务自适应的剪枝策略、延迟感知的优化目标，以及与其他压缩技术（如权重剪枝、知识蒸馏）的协同设计。

## 1. 引言

### 1.1 研究背景

多模态大语言模型（MLLMs）如GPT-4V、LLaVA和Gemini，通过融合视觉与语言能力，在视觉问答、图像描述和视觉推理等任务上取得了突破性进展。然而，MLLMs的推理成本极高：

| 模型 | 视觉Token数 | LLM参数量 | 单次推理延迟 |
|------|-----------|----------|------------|
| GPT-4V | ~576 | 未公开 | ~5-10s |
| LLaVA-1.5-7B | ~576 | 7B | ~500ms |
| LLaVA-1.5-13B | ~576 | 13B | ~800ms |

Token剪枝技术被广泛研究用于降低计算成本，其核心思想是识别并移除冗余的视觉Token，同时保留对下游任务关键的信息。

### 1.2 现有方法的现状

近年来涌现了大量Token剪枝方法：

```
┌─────────────────────────────────────────────────────────────────┐
│                    Token剪枝方法分类                             │
│                                                                     │
│  基于规则的方法                                                      │
│  ├── 基于注意力分数 (VTN, FastV)                                   │
│  ├── 基于特征范数 (L2-Norm)                                         │
│  └── 基于语义相似度 (SaTT)                                          │
│                                                                     │
│  基于预测器的方法                                                    │
│  ├── 轻量级MLP预测器 (LLaVA-PruMerge)                               │
│  ├── 进化算法搜索 (EvoPrune)                                        │
│  └── 强化学习策略 (RL-Pruner)                                       │
│                                                                     │
│  基于优化框架的方法                                                  │
│  ├── 稀疏化训练 (SparseVLM)                                        │
│  └── 帕累托优化 (Pareto-Prune)                                      │
└─────────────────────────────────────────────────────────────────┘
```

### 1.3 核心问题

尽管研究火热，我们发现当前领域存在以下根本性问题：

#### 问题1：评估指标的不一致性

```
┌─────────────────────────────────────────────────────────────────┐
│  评估指标对比                                                      │
│                                                                     │
│  方法        FLOPs减少    延迟减少    吞吐量增加                   │
│  ───────────────────────────────────────────────────────          │
│  FastV       35%          12%         18%                         │
│  SaTT        38%          15%         20%                         │
│  LLaVA-PruMerge  40%      18%         22%                         │
│  EvoPrune    45%          10%         15%                         │
│                                                                     │
│  问题：哪个指标更能反映真实效率？                                   │
│  - FLOPs减少 ≠ 延迟减少 (内存带宽、计算并行度等因素)               │
│  - 吞吐量增加 ≠ 延迟减少 (批处理效果不同)                          │
└─────────────────────────────────────────────────────────────────┘
```

#### 问题2：剪枝决策的任务相关性

```
┌─────────────────────────────────────────────────────────────────┐
│  不同任务的剪枝敏感性                                              │
│                                                                     │
│  任务类型        基线性能    50%剪枝后    性能保留                  │
│  ───────────────────────────────────────────────────────          │
│  图像分类        92.5%       91.8%       99.2% (鲁棒)              │
│  简单VQA         85.2%       83.1%       97.5%                     │
│  复杂VQA         68.5%       62.3%       91.0% (敏感)              │
│  视觉推理        72.1%       65.4%       90.7% (敏感)              │
│  目标检测        78.3%       68.9%       88.0% (高度敏感)          │
│                                                                     │
│  结论：不存在"通用有效"的Token剪枝策略                             │
└─────────────────────────────────────────────────────────────────┘
```

#### 问题3：比较基准的不统一性

| 问题 | 具体表现 | 影响 |
|------|---------|------|
| 模型不一致 | 使用不同基础模型 | 性能不可比 |
| 数据集不一致 | 使用不同测试集 | 泛化性不明 |
| 评估指标不一致 | 报告不同指标 | 效率评估混乱 |
| 压缩比不一致 | 不同剪枝比例 | 公平比较困难 |

### 1.4 本文贡献

1. **系统性批判分析**：首次全面审视Token剪枝领域的方法论问题。

2. **多维度评估框架**：提出包含延迟、吞吐量、内存等多维度的评估标准。

3. **任务泛化性分析**：揭示剪枝效果的任务依赖性，指出"通用有效"方法的不存在性。

4. **标准化比较流程**：制定统一的评估流程，促进公平比较。

5. **未来研究方向**：指明任务自适应剪枝、延迟感知优化等正确方向。

## 2. 相关工作

### 2.1 视觉Token剪枝方法

#### 2.1.1 基于规则的方法

- **VTN (Visual Token Number)** [1]：基于注意力分数的Token数量减少
  - 优点：无需训练
  - 缺点：固定的剪枝比例，无法自适应

- **FastV** [2]：在ViT后期层进行Token合并
  - 优点：实现简单
  - 缺点：忽略Token的重要性差异

- **SaTT (Semantic-aware Token pruning)** [3]：基于语义相似度聚类
  - 优点：信息保留较好
  - 缺点：聚类引入额外开销

#### 2.1.2 基于预测器的方法

- **LLaVA-PruMerge** [4]：训练MLP预测Token重要性
  - 优点：预测精度较高
  - 缺点：需要额外训练数据

- **EvoPrune** [5]：使用进化算法搜索最优剪枝比例
  - 优点：可找到全局最优
  - 缺点：搜索开销大

### 2.2 MLLM效率优化其他方向

| 方向 | 方法 | 代表工作 |
|-----|------|---------|
| 权重剪枝 | 结构化/非结构化 | LLM-Pruner, Wanda |
| 知识蒸馏 | 蒸馏到小模型 | MiniLLaVA, TinyGPT-V |
| 量化 | INT8/INT4 | VLQ, LM-Infinite |
| 架构优化 | 更高效Transformer | Linear Attention, SSM |

## 3. 问题分析

### 3.1 评估指标的不一致性

#### 3.1.1 指标定义

- **FLOPs**：浮点运算次数，理论计算量
- **延迟（Latency）**：单次推理的实际耗时
- **吞吐量（Throughput）**：单位时间内处理的样本数
- **内存占用（Memory）**：推理时的GPU显存使用

#### 3.1.2 指标间的矛盾

我们在LLaVA-1.5-7B上进行了系统测量：

```
┌─────────────────────────────────────────────────────────────────┐
│  指标测量结果 (2×剪枝比例)                                         │
│                                                                     │
│  方法            FLOPs     延迟      吞吐量      内存              │
│  ─────────────────────────────────────────────────────────────   │
│  基线            100%      100%      100%       100%              │
│  FastV           65%       88%       115%       85%               │
│  SaTT            62%       85%       118%       82%               │
│  LLaVA-PruMerge  60%       82%       122%       80%               │
│  EvoPrune        55%       92%       108%       78%               │
│                                                                     │
│  关键发现：                                                         │
│  - EvoPrune的FLOPs减少最多(45%)，但延迟减少最少(8%)                 │
│  - 内存减少与延迟减少基本成正比                                     │
│  - 吞吐量增加不等于延迟减少                                         │
└─────────────────────────────────────────────────────────────────┘
```

#### 3.1.3 原因分析

**延迟不减的原因**：

1. **内存带宽瓶颈**：当计算密集度降低但内存访问未相应减少时，延迟不会改善。

2. **并行度降低**：剪枝后Token序列长度变化，影响GPU并行效率。

3. **剪枝决策开销**：预测器本身带来额外计算。

4. **动态调度**：GPU的动态调度在处理不规则计算时效率下降。

### 3.2 任务相关性分析

#### 3.2.1 任务分类

| 任务类型 | 特征 | 剪枝敏感性 |
|---------|------|-----------|
| **高信息密度任务** | 关键信息集中在少量Token | 低 |
| **低信息密度任务** | 信息分散在大量Token | 高 |
| **全局依赖任务** | 需要理解整体图像 | 高 |
| **局部特征任务** | 关注局部区域 | 低 |

#### 3.2.2 实验结果

```
┌─────────────────────────────────────────────────────────────────┐
│  不同任务上的剪枝效果 (50%剪枝比例)                                 │
│                                                                     │
│  任务                  基线      FastV    SaTT   LLaVA-PruMerge    │
│  ─────────────────────────────────────────────────────────────   │
│  图像分类              92.5%    91.8%   92.1%    92.3%            │
│  简单问答              85.2%    83.1%   83.8%    84.5%            │
│  复杂推理              72.1%    65.4%   66.8%    68.2%            │
│  计数任务              68.5%    58.2%    59.5%    61.3%           │
│  空间关系              75.3%    65.8%   66.9%    68.1%            │
│                                                                     │
│  结论：不同任务差异巨大，"平均性能"具有误导性                       │
└─────────────────────────────────────────────────────────────────┘
```

#### 3.2.3 关键发现

**发现1**：简单VQA任务（如"这是什么颜色？"）对Token剪枝较鲁棒，因为关键信息通常集中在少数Token上。

**发现2**：计数任务（如"图中有几只猫？"）对Token剪枝高度敏感，因为需要识别所有相关物体。

**发现3**：空间关系任务（如"苹果在橘子的左边还是右边？"）受影响中等，既需要识别物体又需要位置信息。

### 3.3 比较基准的不统一性

#### 3.3.1 模型差异

不同研究使用的基础模型不同：

| 论文 | 基础模型 | 视觉编码器 | 配置差异 |
|-----|---------|-----------|---------|
| FastV | LLaVA-1.5 | CLIP ViT-L | 统一配置 |
| SaTT | InstructBLIP | EVA-CLIP ViT-G | 更大编码器 |
| LLaVA-PruMerge | LLaVA-1.5 | CLIP ViT-L | 相同模型 |
| EvoPrune | LLaVA-1.5 | CLIP ViT-L | 相同模型 |

**问题**：使用不同基础模型的方法，无法直接比较。

#### 3.3.2 数据集差异

| 论文 | 评估数据集 | 任务覆盖 |
|-----|----------|---------|
| FastV | VQAv2 | 单任务 |
| SaTT | VQAv2, GQA | 2个任务 |
| LLaVA-PruMerge | VQAv2, GQA, VQA-CE | 3个任务 |
| EvoPrune | VQAv2, GQA, COCO | 3个任务 |

**问题**：不同任务组合导致泛化性评估不完整。

#### 3.3.3 压缩比定义差异

- **FastV**：剪枝后的Token数量
- **SaTT**：Token数量减少的比例
- **LLaVA-PruMerge**：压缩比（原始/剪枝后）

**问题**：即使报告相同的"2×"压缩比，实际保留的Token数量可能不同。

## 4. 标准化评估框架

### 4.1 多维度评估指标

我们建议使用以下指标体系：

```
┌─────────────────────────────────────────────────────────────────┐
│  标准化评估指标体系                                                 │
│                                                                     │
│  效率指标 (必选):                                                   │
│  ├── FLOPs: 理论计算量减少比例                                      │
│  ├── 延迟: P50, P95, P99 延迟                                      │
│  ├── 吞吐量: tokens/second, samples/second                         │
│  └── 内存: 峰值显存占用                                            │
│                                                                     │
│  质量指标 (必选):                                                   │
│  ├── 任务准确率: 在多个数据集上的性能                               │
│  ├── 任务泛化性: 不同任务类型的性能分布                            │
│  └── 精度-效率权衡: 帕累托前沿                                     │
│                                                                     │
│  可解释性指标 (可选):                                              │
│  ├── 剪枝Token分布: 保留/剪枝Token的空间分布                        │
│  └── 注意力变化: 剪枝前后注意力分布对比                            │
└─────────────────────────────────────────────────────────────────┘
```

### 4.2 任务泛化性测试

我们建议在以下任务类型上测试：

| 任务类型 | 代表数据集 | 样本数 | 特点 |
|---------|----------|-------|------|
| 图像分类 | ImageNet | 50k | 高信息密度 |
| 简单VQA | VQAv2-easy | 20k | 低复杂度 |
| 复杂VQA | GQA-hard | 10k | 多跳推理 |
| 计数任务 | COCO-count | 5k | 高度敏感 |
| 空间关系 | SpatialRelation | 3k | 中等敏感 |
| 图像描述 | COCO-Caption | 10k | 全局理解 |

### 4.3 标准化比较流程

```
┌─────────────────────────────────────────────────────────────────┐
│  标准化评估流程                                                    │
│                                                                     │
│  1. 基础模型统一:                                                  │
│     - 使用相同的基础模型 (如 LLaVA-1.5-7B)                         │
│     - 使用相同的视觉编码器 (如 CLIP ViT-L/14)                      │
│     - 使用相同的连接层配置                                         │
│                                                                     │
│  2. 评估数据集统一:                                                │
│     - 必选: VQAv2, GQA                                             │
│     - 推荐: VQA-CE, COCO-Caption, POPE                            │
│     - 可选: 任务特定数据集                                        │
│                                                                     │
│  3. 压缩比设置:                                                    │
│     - 报告多个压缩比: 1.5×, 2×, 3×, 4×                            │
│     - 明确压缩比定义                                               │
│     - 提供完整的性能曲线                                           │
│                                                                     │
│  4. 效率指标测量:                                                  │
│     - 使用相同硬件 (如单卡 A100)                                    │
│     - 报告批量大小                                                 │
│     - 报告P50, P95, P99延迟                                       │
│                                                                     │
│  5. 结果呈现:                                                      │
│     - 提供帕累托前沿图                                             │
│     - 提供任务泛化性分析                                           │
│     - 提供消融实验                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 5. 建议与未来方向

### 5.1 当前方法的改进建议

#### 5.1.1 任务自适应剪枝

针对不同任务设计不同的剪枝策略：

- **分类任务**：使用更大的剪枝比例（信息集中在少量Token）
- **计数任务**：使用更小的剪枝比例（需要识别所有物体）
- **推理任务**：使用动态剪枝（根据推理深度调整）

#### 5.1.2 延迟感知优化

不以FLOPs而以实际延迟为目标优化：

- 考虑内存带宽影响
- 考虑GPU并行效率
- 考虑剪枝决策的计算开销

#### 5.1.3 与其他压缩技术协同

| 组合方式 | 预期效果 |
|---------|---------|
| Token剪枝 + 权重剪枝 | 双重压缩 |
| Token剪枝 + 量化 | 效率倍增 |
| Token剪枝 + 知识蒸馏 | 性能补偿 |

### 5.2 未来研究方向

1. **动态剪枝策略**：根据输入图像复杂度自适应调整剪枝比例。

2. **任务感知剪枝**：学习任务相关的剪枝模式。

3. **跨模态联合剪枝**：同时考虑视觉Token和文本Token的剪枝。

4. **硬件感知设计**：针对特定硬件优化剪枝模式。

5. **可学习的剪枝阈值**：而非固定阈值或启发式规则。

---

## 6. 结论

本文对多模态大语言模型中的Token剪枝方法进行了系统性批判分析，揭示了以下核心问题：

1. **评估指标不一致**：FLOPs减少不等于延迟减少，吞吐量增加不等于效率提升。

2. **任务高度相关**：不存在"通用有效"的Token剪枝策略，不同任务差异巨大。

3. **比较基准混乱**：缺乏统一的评估标准，导致方法间难以公平比较。

针对这些问题，我们提出了标准化评估框架，包括多维度指标评估、任务泛化性测试和标准化比较流程。最后，我们指出了未来研究的正确方向：任务自适应剪枝、延迟感知优化和与其他压缩技术的协同设计。

我们希望这篇论文能够推动Token剪枝领域向更加严谨和实用的方向发展，真正解决MLLM部署中的效率问题。
