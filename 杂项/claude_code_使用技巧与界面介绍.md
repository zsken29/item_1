# Claude Code 使用技巧与界面介绍

更新日期：2026-05-04  
适用对象：已经会基本使用命令行，希望把 Claude Code 用成日常开发助手的开发者。

## 1. Claude Code 是什么

Claude Code 是 Anthropic 提供的智能编码工具。它可以运行在终端、IDE、桌面端和浏览器中，读取和修改本地项目文件、执行命令、解释代码、修复错误、生成测试、协助提交 Git 变更，并能通过 IDE 集成、MCP、Hooks、Commands、Skills 等能力扩展工作流。

它更像一个“在仓库里工作的结对开发者”，而不是单纯的聊天窗口。高效使用的关键不是一次性问大而空的问题，而是把目标、约束、验证方式和代码上下文交代清楚，让它在项目里完成可验证的改动。

## 2. 主要界面入口

### 2.1 终端交互界面

这是 Claude Code 的核心界面。常见启动方式：

```bash
claude
```

在项目根目录运行后，Claude Code 会进入交互式会话。你可以直接用自然语言描述任务，例如：

```text
阅读这个项目的入口文件，解释启动流程
```

```text
修复当前测试失败的问题，并运行相关测试验证
```

终端界面通常包含这些交互元素：

- 输入区：你输入任务、问题或后续指令。
- 模型回复区：Claude 展示分析、计划、代码变更说明和命令结果摘要。
- 工具调用区：当 Claude 读取文件、搜索代码、运行测试或编辑文件时，会显示相应操作。
- 权限确认：涉及文件修改、命令执行、网络访问或高风险操作时，可能要求你确认。
- 上下文延续：同一会话里，Claude 会记住此前讨论、已读文件和你的偏好。

### 2.2 一次性命令模式

如果你只想让 Claude 执行一个明确任务，可以用 `-p` 传入提示词：

```bash
claude -p "总结这个仓库的目录结构和主要模块"
```

这种模式适合脚本化、自动化或快速查询。对于需要多轮确认的复杂修改，交互式会话更合适。

### 2.3 IDE 集成界面

Claude Code 支持与 VS Code、JetBrains 等 IDE 配合使用。IDE 集成的价值主要在于：

- 当前打开文件、选中文本、诊断信息能更自然地进入上下文。
- Claude 生成或修改代码后，你可以在熟悉的编辑器里审阅 diff。
- 适合“我正在看这一段代码，请解释/重构/补测试”的场景。

典型用法是：先在 IDE 打开项目，再从集成终端或插件入口启动 Claude Code。复杂改动仍建议让 Claude 明确列出变更文件、验证命令和风险点。

### 2.4 桌面端和浏览器界面

桌面端和浏览器界面适合更长周期的任务，例如并行跑多个任务、可视化审阅 diff、使用集成终端和文件编辑器、预览应用、安排定时任务，或在离开本机环境时继续查看进展。相比纯终端，它们更强调任务管理和视觉审查；相比 IDE，它们对具体编辑器依赖更少。

使用建议：

- 本地精细修改优先用终端或 IDE。
- 长时间运行、需要回头查看的任务可以用 Web 或 Desktop。
- 重要代码合并前仍要回到 Git diff、测试和代码审查流程。

### 2.5 Git 与代码审查界面

Claude Code 可以辅助理解 diff、生成提交信息、检查潜在问题。常见请求：

```text
审查当前未提交改动，优先指出 bug、回归风险和缺失测试
```

```text
根据当前 staged diff 写一个简洁的 commit message
```

```text
解释最近一次提交改了什么，是否有兼容性风险
```

做代码审查时，最好要求 Claude 以“问题优先”的方式输出，而不是先总结优点。这样更容易发现真实风险。

## 3. 常用命令与交互能力

### 3.1 Commands 与 Slash Commands

Commands 是以 `/` 开头的会话内命令，用于控制会话、查看状态或触发特定动作。常见类型包括：

- `/help`：查看可用帮助。
- `/clear`：清理当前会话上下文，适合开始新任务。
- `/compact`：压缩长会话上下文，保留关键进展。
- `/model`：查看或切换模型。
- `/status`：查看当前会话、配置或连接状态。
- `/permissions`：管理工具权限。
- `/memory`：查看或编辑记忆相关内容。

不同版本、平台、订阅计划和配置下可用命令可能略有差异，实际以 `/help` 或输入 `/` 后出现的菜单为准。自定义 Slash Commands 现在主要通过 Skills 机制组织，也可以继续兼容旧的 `.claude/commands/` 文件。

### 3.2 CLI 常用参数

常见命令形式：

```bash
claude
claude -p "你的任务"
claude --help
claude --version
```

实用习惯：

- 不确定参数时先运行 `claude --help`。
- 自动化任务优先使用 `claude -p`。
- 多轮开发、调试、重构优先使用交互式 `claude`。

### 3.3 文件与代码搜索

Claude Code 会根据任务自动读取文件、搜索引用、检查测试。你也可以明确要求它采用某种方式：

```text
先用 rg 搜索所有调用点，再决定最小改动范围
```

```text
只修改 parser 相关文件，不要碰 UI 和配置
```

```text
先解释计划，等我确认后再改代码
```

这种约束能显著降低无关改动。

## 4. 推荐工作流

### 4.1 阅读陌生项目

推荐提示词：

```text
请先阅读项目结构和 README，找出入口文件、核心模块、测试方式和主要依赖。不要修改文件，给我一份简洁的架构说明。
```

更进一步：

```text
按“启动流程、数据流、核心类/函数、外部依赖、潜在风险”总结这个项目。
```

### 4.2 修复 Bug

推荐流程：

1. 复现问题。
2. 定位相关代码。
3. 解释根因。
4. 做最小修改。
5. 添加或更新测试。
6. 运行验证命令。

推荐提示词：

```text
这个 bug 的现象是：……
请先尝试复现，定位根因后再修改。修改范围尽量小，最后运行相关测试并说明验证结果。
```

### 4.3 新增功能

推荐提示词：

```text
新增功能：……
约束：
- 保持现有代码风格
- 不引入新依赖，除非必要并说明原因
- 补充最小必要测试
- 完成后列出改动文件和验证命令
```

如果功能较大，先让 Claude 拆分任务：

```text
先不要改代码。请阅读相关模块，给出实现方案、受影响文件、测试计划和风险点。
```

### 4.4 重构

重构任务最容易失控，建议明确边界：

```text
只重构认证模块内部结构，不改变外部 API 和行为。请先列出当前行为假设，然后分小步修改，每一步都保持测试可通过。
```

有效约束包括：

- 不改变公共接口。
- 不移动无关文件。
- 不格式化整个仓库。
- 不引入新抽象，除非能减少明确重复或复杂度。
- 保留现有测试语义。

### 4.5 写测试

推荐提示词：

```text
为这个模块补测试。优先覆盖边界条件、错误分支和之前容易回归的行为。不要为了测试而改生产代码，除非发现可测试性问题并先说明。
```

对于失败测试：

```text
运行相关测试，定位失败原因。区分是测试断言过期、实现有 bug，还是环境问题。
```

## 5. 提示词技巧

### 5.1 明确目标和完成标准

低效写法：

```text
优化一下这个项目
```

高效写法：

```text
优化数据导入流程的性能。目标是减少重复数据库查询，不改变导入结果。请先找出热点路径，再给出最小修改方案，并用现有测试验证。
```

### 5.2 给出约束

常用约束：

- “不要修改公共 API。”
- “不要新增依赖。”
- “只处理这个错误，不做顺手重构。”
- “先给计划，等我确认。”
- “直接实现，完成后运行测试。”
- “如果测试跑不起来，说明原因和替代验证。”

### 5.3 要求 Claude 解释证据

让 Claude 把判断建立在代码和命令结果上：

```text
请指出你依据了哪些文件和函数，不要只给结论。
```

```text
如果你推断了某个行为，请标注这是推断还是代码中已经明确存在。
```

### 5.4 控制输出格式

适合开发任务的输出格式：

```text
完成后请按以下格式回复：
1. 改了什么
2. 涉及文件
3. 验证命令和结果
4. 剩余风险
```

对于代码审查：

```text
请按严重程度列出问题。每个问题包含文件、行号、原因、可能影响和建议修复方式。
```

## 6. 记忆与项目上下文

Claude Code 可以使用项目级或用户级记忆来保存长期偏好和项目说明。最常见的是 `CLAUDE.md` 文件。

建议在项目根目录维护一份简洁的 `CLAUDE.md`，内容包括：

```markdown
# Project Notes

## Commands
- Install: ...
- Test: ...
- Lint: ...
- Run: ...

## Code Style
- ...

## Architecture
- ...

## Constraints
- Do not ...
```

好的 `CLAUDE.md` 应该短、准、可执行。不要把大量过期文档塞进去；Claude 每次加载无关内容都会浪费上下文。

适合写入记忆的内容：

- 常用测试命令。
- 项目目录约定。
- 代码风格偏好。
- 不应触碰的生成文件。
- 部署、构建或环境注意事项。

不适合写入记忆的内容：

- 临时任务。
- 密钥和凭证。
- 大段日志。
- 已经过期的设计争论。

## 7. 权限、安全与确认

Claude Code 能执行命令和修改文件，因此权限管理很重要。

建议原则：

- 对读取文件、搜索代码可以较宽松。
- 对写文件、安装依赖、删除文件、执行迁移、推送代码要谨慎。
- 不要让 Claude 自动处理密钥、令牌、生产数据库凭证。
- 涉及破坏性命令时，要求它先解释目标路径和影响范围。

实用提示词：

```text
任何删除、移动、重命名、安装依赖、数据库迁移或 git push 操作前，都必须先说明计划并等待我确认。
```

```text
不要读取 .env、密钥文件或任何包含 token 的文件。
```

## 8. Hooks 与自动化

Hooks 可以在 Claude Code 的特定生命周期事件上执行预设命令，例如在编辑后运行格式化、在提交前运行测试、在会话结束时生成摘要等。

适合使用 Hooks 的场景：

- 修改 Python 文件后自动运行 formatter。
- 修改前端文件后自动运行 lint。
- 会话结束时记录变更摘要。
- 阻止某些高风险命令。

使用建议：

- Hooks 应该短小、确定、可重复执行。
- 不要在 Hook 中隐藏复杂业务逻辑。
- 对耗时很长的测试，最好让 Claude 显式运行，而不是每次自动触发。
- 团队项目中要把 Hook 行为写清楚，避免别人困惑。

## 9. MCP 与外部工具

MCP 可以把外部系统接入 Claude Code，例如 GitHub、数据库、文档系统、设计工具或内部服务。接入后，Claude 可以通过受控工具读取或操作这些系统。

适合 MCP 的场景：

- 查 GitHub issue、PR、CI 状态。
- 查询内部文档。
- 访问设计稿或产品需求。
- 调用只读数据库查询。

使用建议：

- 优先只读权限起步。
- 为每个工具限制清晰用途。
- 对写操作保留人工确认。
- 不要把生产级高权限系统直接暴露给自动化流程。

## 10. 常见高效场景

### 10.1 解释错误日志

```text
这是报错日志：……
请定位最可能的根因，指出相关文件，并给出验证方式。
```

### 10.2 生成迁移说明

```text
比较当前分支和 main，写一份面向团队的变更说明，重点说明行为变化、配置变化和升级注意事项。
```

### 10.3 生成 PR 描述

```text
根据当前 diff 生成 PR 描述，包括背景、改动点、测试结果和风险。
```

### 10.4 梳理技术债

```text
阅读这个模块，找出最值得优先处理的 5 个技术债。每项说明影响、证据和建议改法。不要直接改代码。
```

### 10.5 调试 CI

```text
根据这段 CI 日志定位失败原因。请区分代码问题、依赖问题和 CI 环境问题，并给出最小修复方案。
```

## 11. 常见误区

### 11.1 一次给太大任务

“重写整个项目”通常效果不好。更好的方式是拆成可验证的小任务，例如先重构一个模块、先补测试、先替换一个接口。

### 11.2 不提供验证标准

如果没有说明怎样算完成，Claude 可能只做到“看起来完成”。最好明确测试命令、目标行为和不能破坏的兼容性。

### 11.3 让 Claude 猜业务规则

业务规则不在代码里时，Claude 只能推断。涉及产品、财务、合规、用户权限等逻辑时，要把规则写清楚。

### 11.4 忽略 diff 审查

Claude 生成的代码仍然需要审查。尤其关注：

- 是否改了无关文件。
- 是否引入新依赖。
- 是否改变公共接口。
- 是否只修了表象，没有覆盖根因。
- 测试是否真的验证了核心行为。

## 12. 推荐个人配置习惯

建议形成一套固定模板：

```text
任务：
约束：
相关文件：
完成标准：
验证命令：
输出格式：
```

示例：

```text
任务：修复用户导入时重复创建记录的问题。
约束：不改变 API；不新增依赖；只改 import 模块。
相关文件：src/importer.py、tests/test_importer.py。
完成标准：相同 email 只创建一次，重复行记录 warning。
验证命令：pytest tests/test_importer.py。
输出格式：说明根因、改动、测试结果、剩余风险。
```

## 13. 快速检查清单

开始任务前：

- 目标是否明确？
- 修改范围是否明确？
- 是否说明不能改什么？
- 是否给出验证方式？
- 是否需要先让 Claude 只读分析？

Claude 完成后：

- 看 diff 是否集中。
- 运行测试或确认 Claude 已运行。
- 检查是否有无关格式化。
- 检查新增依赖和配置变化。
- 要求说明剩余风险。
